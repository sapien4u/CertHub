Module 6 · Developer Productivity

SQL Server & data

Copilot helps with queries, stored procedures, indexing suggestions, and EF Core mappings — but SQL is where correctness, performance, and injection review matter most.

Query authoring & optimization

// Prompt
Rewrite this query to be sargable and avoid a scan: it filters on
YEAR(OrderDate) = 2025. Suggest a supporting index.

-- Sargable rewrite (uses an index on OrderDate)
SELECT OrderId, Total
FROM Orders
WHERE OrderDate >= '2025-01-01' AND OrderDate < '2026-01-01';
★Best Practice

Ask Copilot to make predicates sargable, to explain the execution plan implications, and to propose indexes — then validate with an actual execution plan. Treat index suggestions as hypotheses to test, not truths.

Stored procedures

Write a stored procedure usp_GetOrdersByCustomer(@CustomerId INT, @FromUtc DATETIME2)
that returns paged orders; use parameters (no dynamic SQL), SET NOCOUNT ON,
and an OFFSET/FETCH page of 50.

Injection safety

⚠Common Mistake

Accepting dynamic SQL built by string concatenation. It's an injection risk. Prefer parameters; if dynamic SQL is unavoidable, use sp_executesql with parameters and review carefully.

EF Core mapping & migrations

Give Copilot your entity and DbContext so generated configurations and migrations match. Always read migrations before applying — they can drop columns or data if intent is misread.

Performance review checklist

  • Sargable predicates; no functions on indexed columns in the WHERE.
  • Right indexes; verify with execution plans, not guesses.
  • Avoid SELECT *; project needed columns.
  • Watch for implicit conversions and parameter sniffing.
✓Exam Tip

For data tasks, Copilot drafts SQL/EF quickly; you must verify correctness against the schema, performance via plans, and injection safety (parameters over concatenation).
