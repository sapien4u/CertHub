Module 6 · Developer Productivity

Azure

Copilot accelerates Azure development — Functions, Key Vault, Storage, Bicep, App Service — but cloud code needs extra care around secrets, identity, and cost.

Azure Functions

// Prompt
.NET 8 isolated-worker Azure Function, HTTP trigger POST /orders:
validate the body, call IOrderService, return 201 with the created id.
Use ILogger and DI; no secrets in code.

Key Vault & managed identity

Steer Copilot toward managed identity and DefaultAzureCredential rather than connection strings or keys.

var client = new SecretClient(new Uri(kvUri), new DefaultAzureCredential());
var secret = await client.GetSecretAsync("Db--ConnectionString");
★Best Practice

Prompt explicitly for managed identity / DefaultAzureCredential and Key Vault references. Reject any suggestion that embeds keys or connection strings in code or config.

Bicep / Infrastructure as Code

Copilot drafts Bicep for common resources; verify SKUs, regions, naming, and that outputs/parameters match your standards.

Write a Bicep module for a Storage account (Standard_LRS, TLS 1.2, no public blob),
with a parameter for name and a diagnostic setting to Log Analytics.
⚠Common Mistake

Accepting IaC that opens public access, uses weak TLS, or hard-codes secrets. Generated infra can carry insecure defaults — review every property.

App Service & configuration

Ask for configuration via App Settings / Key Vault references, health checks, and appropriate scaling settings — then confirm against your landing-zone policies.

Best practices

★Best Practice
  • Default to identity-based auth (managed identity) over keys.
  • Keep secrets in Key Vault; never in code, and never pasted into prompts.
  • Review generated IaC for security defaults (TLS, public access, RBAC) and cost (SKU/tier).
  • Validate region/quota assumptions before deploying.
✓Exam Tip

The Responsible-AI themes reappear in cloud form: generated Azure code isn't pre-secured. You own the review of identity, secret handling, network exposure, and cost.
