Module 3 · Prompt Engineering

Prompting principles

Prompt engineering is the practice of communicating intent so Copilot produces accurate, relevant output. For code, a few durable principles do most of the work.

The four principles

  1. Be specific. State the language, framework/version, inputs, outputs, and constraints. "Angular 18 standalone component using signals" beats "an Angular component."
  2. Give context. Open or reference the relevant types and examples so the model sees your domain, not a generic one.
  3. Show the shape. Provide a signature, an example input/output, or an existing pattern to imitate.
  4. Iterate. Treat the first response as a draft; refine with follow-ups rather than restarting.
≡Example

Vague: "write a function to process orders" Specific: "Write a C# method decimal CalculateTotal(Order order) that sums Quantity * UnitPrice * (1 + TaxRate) per line, rounds to 2dp, and throws ArgumentException if any quantity is negative."

Anatomy of a strong prompt

  • Goal — what you want produced.
  • Context — the relevant code/types (referenced, not pasted when possible).
  • Constraints — versions, patterns, performance, security, style.
  • Format — the exact output shape (a method, a test class, a diff).
Goal: Add pagination to the users endpoint.
Context: #file:UsersController.cs, #file:User.cs
Constraints: .NET 8 minimal API, page & pageSize query params, max pageSize 100,
             return X-Total-Count header, keep existing filtering.
Format: full updated endpoint plus an xUnit test for the max-pageSize guard.

Good vs poor prompts

Poor Better
"Fix this" "This throws NRE when order.Lines is null; guard it and add a test."
"Make it faster" "This LINQ runs per-row queries (N+1); rewrite to a single EF Core query."
"Write tests" "Write xUnit tests for empty, boundary, and negative-quantity cases."
★Best Practice

Front-load the most important constraint. If security or a specific API matters most, say so first — models weight early, explicit instructions heavily.

⚠Common Mistake

Assuming Copilot shares your unstated assumptions (framework version, naming conventions, which library). If it isn't in the prompt, instructions, or visible context, it's a guess.

✓Exam Tip

When a question asks why Copilot's output was generic or wrong, the best answer is usually insufficient or unclear context/prompt, not a model defect.
