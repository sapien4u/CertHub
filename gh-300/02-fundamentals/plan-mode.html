Module 2 · Copilot Fundamentals

Plan Mode

Plan Mode is a Copilot working mode that focuses on thinking before doing. Instead of immediately editing code, Copilot first produces a structured, reviewable plan — the steps, files, and approach it intends to take — which you refine and approve before any changes are made.

Why Plan Mode exists

Agent mode is powerful but can jump straight into edits. For larger or riskier tasks, you often want to agree on the approach first. Plan Mode separates planning from execution:

  1. You describe the goal.
  2. Copilot proposes a step-by-step plan (what it will change and why).
  3. You edit, reorder, or remove steps.
  4. You approve, and execution proceeds (typically handed to agent mode).

When to use it

Scenario Mode
One suggestion while typing Inline completion
Quick local change across known files Edit mode
"Just do it" multi-step task Agent mode
Complex/unfamiliar task where the approach matters Plan Mode

Plan Mode shines for migrations, refactors that touch many layers, or work in an unfamiliar codebase where you want to validate the strategy before code changes.

Example

Goal: "Migrate the OrderService from synchronous EF Core calls to async,
including controllers, repository, and tests."

Plan Mode output (you review before running):
1. Add async EF Core methods to IOrderRepository and implementation.
2. Thread CancellationToken through OrderService methods.
3. Update OrderController actions to await and accept CancellationToken.
4. Update unit tests to async and add a cancellation test.
5. Run build + tests; fix failures.
★Best Practice

Treat the plan as a contract. Tighten scope, remove steps you'll do yourself, and add explicit constraints (naming, patterns, "don't touch the public API") before approving execution.

✓Exam Tip

On the exam, match the intent: "review and agree the approach before any edits" → Plan Mode; "autonomously carry out the steps" → agent mode.

⚠Common Mistake

Approving a plan without reading it, then being surprised by broad changes. The plan is the cheapest place to catch a wrong approach — read it.

Limitations

  • A plan is only as good as the context and constraints you give it.
  • You still review and approve the resulting edits, not just the plan.
  • Availability varies by surface (IDE) and plan tier.
