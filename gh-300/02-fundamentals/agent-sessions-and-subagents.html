Module 2 · Copilot Fundamentals

Agent sessions & sub-agents

As agent mode has matured, GitHub Copilot organizes autonomous work into agent sessions and can delegate parts of a task to sub-agents. Understanding these helps you manage long-running work and keep context usage efficient.

Agent sessions

An agent session is a persistent, trackable unit of autonomous work. Rather than a single request/response, a session represents Copilot working toward a goal over multiple steps — planning, editing, running tools/tests, and iterating — while you monitor and approve.

  • Sessions can be started, paused, resumed, and reviewed.
  • Each session keeps its own working context and history of actions.
  • You can run work locally (in the IDE) or delegate it to run asynchronously (for example, the Copilot coding agent working on an assigned issue and opening a pull request).

Sub-agents

For complex tasks, an agent can delegate a focused piece of work to a sub-agent. A sub-agent handles a narrower objective with its own scoped context, then returns a result to the parent agent.

Why this matters:

  • Context efficiency — the main session isn't polluted with all the detail of every subtask; each sub-agent works within a smaller, relevant context window.
  • Separation of concerns — e.g., one sub-agent explores and summarizes the codebase while the parent focuses on implementing the change.
  • Better results on large tasks — narrower scope reduces the chance the model loses track of the goal.
Parent agent goal: "Add rate limiting to the public API and document it."
  ├─ Sub-agent A: locate all public endpoints and summarize the middleware pipeline
  ├─ Sub-agent B: draft the rate-limiting middleware + tests
  └─ Parent: integrate, run tests, update docs, open PR

Delegating tasks

You can assign work to Copilot the way you'd assign it to a teammate — for example, assigning a GitHub issue to the Copilot coding agent. It works in a session, pushes a branch, and opens a pull request for human review.

★Best Practice

Give a delegated agent the same clarity you'd give a colleague: a precise goal, acceptance criteria, constraints, and pointers to relevant files. Then review the PR it produces like any other.

✓Exam Tip

Exam cue: "delegate a subtask with its own scoped context to optimize the context window" → sub-agent; "a resumable, trackable unit of autonomous work" → agent session.

⚠Common Mistake

Assuming delegation removes the review step. A delegated agent still produces changes that require human review and approval before merge.

Limitations

  • Autonomy increases the surface area for mistakes; larger diffs are harder to review.
  • Sessions and sub-agents consume usage and depend on plan/surface availability.
  • You remain accountable for everything a session or sub-agent produces.
