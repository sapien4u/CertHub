Module 2 · Copilot Fundamentals

Custom instructions

Custom instructions are always-on guidance that Copilot applies automatically to your requests, so you don't have to restate standing preferences in every prompt. They're the primary way to make Copilot follow team conventions consistently.

Where they live

  • Repository custom instructions — a .github/copilot-instructions.md file committed to the repo. It applies to Copilot Chat requests in that repository for everyone.
  • Path-specific instructions — *.instructions.md files with an applyTo glob, so different rules apply to different parts of the codebase.
  • Personal instructions — user-level preferences that travel with you across repositories (surface-dependent).
<!-- .github/copilot-instructions.md -->
# Project conventions for Copilot
- Target **.NET 8** and **C# 12**; prefer records for DTOs.
- Use **async/await** end-to-end; always accept a CancellationToken.
- Angular code uses **standalone components and signals** (Angular 18).
- Write **xUnit** tests with Arrange/Act/Assert and FluentAssertions.
- Never log secrets or PII; read config from Azure Key Vault.

What to put in them

  • Language/framework versions and idioms.
  • Naming, formatting, and architectural patterns.
  • Testing framework and style.
  • Security and logging rules (e.g., "never hard-code secrets").
  • Things to avoid (deprecated APIs, banned packages).

Best practices

★Best Practice
  • Keep instructions short, specific, and stable — they consume context on every request.
  • Commit them so the whole team benefits and they evolve via PRs.
  • Use path-specific files for monorepos with mixed stacks.
  • State rules as imperatives ("use X", "never do Y").

Limitations

  • Instructions are guidance, not hard guarantees — Copilot can still deviate.
  • Overly long instructions waste the context budget and dilute focus.
  • They don't replace review, tests, or linting.
⚠Common Mistake

Writing a huge, vague instructions file ("write good, clean, secure code"). Non-specific guidance changes little. Prefer concrete, checkable rules.

✓Exam Tip

Contrast the three levers: custom instructions = always-on conventions; prompt files = reusable task prompts; inline prompt = one-off request. The exam likes to test which mechanism fits which need.
