Module 2 · Copilot Fundamentals

Tests, docs, refactoring & debugging

Some of Copilot's highest-value everyday tasks are generating tests, writing documentation, refactoring safely, and helping debug. These map to the exam's "developer use cases" and "testing" domains.

Test generation

Copilot drafts unit tests from your code and intent — a strong fit for xUnit/NUnit in .NET and Jasmine/Jest in Angular. Ask for specific cases, not just "write tests."

/tests Generate xUnit tests for OrderService.CalculateTotal covering:
empty order, single line, tax rounding to 2dp, and a negative quantity guard.
Use FluentAssertions.
★Best Practice

Prompt for edge cases and failure modes explicitly (nulls, empty collections, boundaries, exceptions). Copilot happily writes the happy path; you must direct it to the risky paths.

Documentation

Copilot generates XML doc comments, README sections, and inline explanations.

/// <summary>Calculates the order total including per-line tax.</summary>
/// <param name="order">The order aggregate; lines must have non-negative quantities.</param>
/// <returns>The tax-inclusive total, rounded to 2 decimal places.</returns>

Refactoring

Use chat or edit mode to restructure code while preserving behavior: extract methods, introduce patterns, modernise syntax (e.g., to C# 12 or Angular signals).

★Best Practice

Refactor behind tests. Generate/confirm tests first, refactor second, then re-run the suite. This turns "does it still work?" into an automatic check.

Debugging

Paste an error, stack trace, or failing test into chat and ask for likely causes and fixes. /fix targets the current problem in context.

/fix This xUnit test throws NullReferenceException on _repo.GetAsync.
Here is the test #selection and the service #file:OrderService.cs.

Limitations

  • Generated tests may assert the current (possibly buggy) behavior rather than the correct behavior — you define correctness.
  • Refactors can subtly change behavior; only tests catch that reliably.
  • Debugging suggestions are hypotheses, not diagnoses.

Common mistakes

⚠Common Mistake
  • Trusting generated tests as proof of correctness. Tests written from the same flawed assumptions can pass while the code is wrong.
  • Refactoring without a test safety net and assuming behavior is preserved.
  • Accepting a "fix" without confirming it addresses the root cause, not just the symptom.
✓Exam Tip

For the testing domain: Copilot accelerates test writing but you must specify coverage (edge cases) and judge correctness. Passing tests generated by Copilot are not, by themselves, evidence that the code meets requirements.
