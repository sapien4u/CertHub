Module 1 · Introduction

How Copilot works

At a high level, Copilot turns your context into a prompt, sends it to a large language model (LLM), and streams back a suggestion that you accept, edit, or reject. Knowing the pipeline explains why context matters so much and why output varies.

The suggestion pipeline

  1. Context gathering. Copilot collects signals from your environment: the current file, the code around your cursor (before and after), other open tabs/files, the file name and language, and — depending on surface and plan — related files, symbols, and repository content.
  2. Prompt assembly. These signals are packaged into a prompt within a bounded context window (a token limit). Not everything can fit, so relevance and proximity matter.
  3. Model inference. The LLM predicts the most likely continuation or answer.
  4. Post-processing & filtering. Results pass through safety systems, including a filter that can block suggestions matching public code, and vulnerability-pattern filtering.
  5. Presentation. You see a completion (ghost text), a chat response, or proposed edits. Nothing is applied without your action.
ℹNote

The context window is finite. If the relevant code is far away, in a closed file, or buried under noise, it may not influence the suggestion. This is the mechanical reason "context engineering" works.

What counts as context

  • Code immediately before and after the cursor (Copilot uses both).
  • Open files / tabs in the editor (neighboring context).
  • File name, language, and project signals.
  • Comments and existing patterns in the file.
  • On richer surfaces/plans: workspace context, repository indexing, chat participants like @workspace, and referenced files.

Why suggestions vary

Because the model is probabilistic and the assembled context changes as you type or open files, the same cursor position can yield different suggestions. This is expected behavior, not a bug.

// With little context, Copilot guesses generically:
public decimal CalculateTotal( /* ? */ )

// With nearby types and a clear comment, it aligns to your domain:
// Sum line totals including tax for an Order aggregate
public decimal CalculateTotal(Order order) =>
    order.Lines.Sum(l => l.Quantity * l.UnitPrice * (1 + l.TaxRate));

Content exclusions and filtering

Administrators can configure content exclusions so that specified files or paths are not used as context or sent to the service. Copilot also applies a duplication detection filter (when enabled) that suppresses suggestions matching public code, and screens for some insecure coding patterns.

✓Exam Tip

Two exam-relevant control points: content exclusions (what Copilot may see) and the duplicate/matching-public-code filter (what Copilot may suggest). Know that both exist and roughly what each does.

★Best Practice

Give Copilot the shortest path to the right context: open the files that define the types you're using, keep intent-revealing comments nearby, and remove distracting or dead code from view when generating.

⚠Common Mistake

Expecting Copilot to "know" your whole repository by default in every surface. Broad repository awareness depends on the surface, plan, and features like indexing — it is not guaranteed everywhere.
