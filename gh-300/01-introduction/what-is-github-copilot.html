Module 1 · Introduction

What is GitHub Copilot

GitHub Copilot is an AI pair programmer that helps you write, understand, and maintain code faster. It integrates directly into the editor and developer workflow — offering inline code completions, a conversational chat assistant, multi-file edits, autonomous agent tasks, terminal help, and pull-request assistance.

For an experienced engineer, the mental model that matters is this: Copilot is a context-driven suggestion engine, not a search engine and not a compiler. It predicts likely code and prose from the context you give it, and you remain the accountable author and reviewer of everything it produces.

Where Copilot fits in your workflow

Copilot spans the full software development lifecycle rather than a single task:

  • Authoring — inline completions and Copilot Chat while writing new code.
  • Understanding — explaining unfamiliar code, dependencies, or stack traces.
  • Changing — edit mode and agent mode for multi-file refactors and features.
  • Reviewing — PR summaries and code review suggestions.
  • Operating — CLI command help, test generation, and documentation.
ℹNote

Copilot is available across popular surfaces: Visual Studio, VS Code, the JetBrains IDEs, Visual Studio for Mac (limited), the GitHub CLI, GitHub Mobile, and github.com. The exact feature set can differ by surface and by plan.

What Copilot is good at

  • Boilerplate and repetitive patterns (DTOs, mappers, CRUD scaffolding).
  • Idiomatic usage of well-known frameworks (EF Core, Angular, xUnit).
  • Explaining and summarising existing code.
  • Drafting unit tests and documentation.
  • Translating intent expressed in comments or chat into a first-draft implementation.

What Copilot is not

  • It is not authoritative. Output can be plausible but wrong (a "hallucination").
  • It is not a substitute for review, testing, or security scanning.
  • It does not guarantee that generated code is free of vulnerabilities or that it matches your architectural standards.
✓Exam Tip

The exam repeatedly tests the boundary between what Copilot does (assist, suggest, accelerate) and what the developer remains responsible for (review, testing, security, licensing, final decisions). When in doubt, the human-in-the-loop answer is usually correct.

★Best Practice

Treat every suggestion as a draft from a talented but junior colleague: fast and often right, but always subject to your review, your tests, and your judgement.

A quick .NET example

Given a comment describing intent, Copilot will draft an implementation:

// Return active users created in the last 30 days, ordered by name
public async Task<List<User>> GetRecentActiveUsersAsync()
{
    var cutoff = DateTime.UtcNow.AddDays(-30);
    return await _db.Users
        .Where(u => u.IsActive && u.CreatedUtc >= cutoff)
        .OrderBy(u => u.Name)
        .ToListAsync();
}

You still verify the field names, the time zone assumptions (UtcNow), and whether an index supports the query — Copilot accelerated the draft, but the engineering decisions are yours.

⚠Common Mistake

Assuming that because code compiles and looks idiomatic, it is correct and secure. Compilation is not validation. Always review logic, edge cases, and security implications.
